#TreeBalancing Benchmark

Порівняльний аналіз стратегій балансування бінарних дерев пошуку (BST).
Проєкт розроблено в рамках лабораторної роботи з програмування (Project 2.Research).

* **Матеріали для дослідження:**
  * [GeeksforGeeks: AVL Tree Data Structure](https://www.geeksforgeeks.org/avl-tree-set-1-insertion/) - використано для реалізації рекурсивного перерахунку висоти та алгоритмів малих/великих поворотів.
  * [Wikipedia: Red–black tree](https://en.wikipedia.org/wiki/Red%E2%80%93black_tree) - теоретична база щодо інваріантів кольорів та 4-х випадків балансування при вставці.
  * [Splay Trees (Sleator & Tarjan, 1985)](https://www.cs.cmu.edu/~sleator/papers/self-adjusting.pdf) - оригінальна стаття розробників структури, використана для розуміння евристики `splay` (zig-zig, zig-zag) та амортизованої складності.

## Результати бенчмаркінгу

Тестування проводилося на випадково згенерованих масивах даних. Час вказано у мікросекундах (us). На великих обсягах даних (N=5000) тестування PBT було пропущено через складність глобальної перебудови O(n), що призводило до зависання.

| Структура | N=100 (Вставка / Видалення) | N=500 (Вставка / Видалення) | N=1000 (Вставка / Видалення) |
| :--- | :--- | :--- | :--- |
| **Perfect Balanced Tree** | 2535 / 1856 | 100364 / 54483 | 1044428 / 188911 |
| **AVL Tree** | 172 / 118 | 517 / 301 | 500 / 565 |
| **Splay Tree** | 40 / 37 | 539 / 205 | 447 / 412 |
| **Red-Black Tree** | 39 / 24 | 172 / 61 | 356 / 135 |

### Аналіз результатів:
1. **Red-Black Tree** показало найкращу стабільність та швидкість на великих масивах даних, оскільки перефарбовування вузлів є значно "дешевшою" операцією, ніж фізичні повороти в пам'яті.
2. **Perfect Balanced Tree** абсолютно не підходить для динамічних систем з частими змінами, що підтверджено експоненційним зростанням часу.
3. **AVL** та **Splay** дерева показали середні результати, очікувані для рандомізованих (а не послідовних) запитів.
